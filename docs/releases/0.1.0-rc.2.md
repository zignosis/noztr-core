# noztr-core 0.1.0-rc.2

Date: 2026-03-22
Release type: breaking rc

## Summary

`0.1.0-rc.2` is the second public release candidate for `noztr-core`.

This RC does not widen the kernel scope, but it does tighten the public API surface after the
first public cut: temporary naming aliases are gone, public error types now consistently prefer
descriptive names inside module namespaces, and the documented verification path now includes a
narrow `zig fmt`-based lint gate.

## Public Highlights

- canonical public symbol names only; temporary compatibility aliases were removed
- public error type names now prefer descriptive module-local names over short-lived numeric
  `NipXXError` forms
- `zig build lint` now runs `zig fmt --check` across the tracked Zig/ZON surface
- the public migration guide and style guide now state the naming rule explicitly

## Breaking Changes

- temporary public naming aliases introduced during the API-naming normalization pass were removed
- public error type names changed across the exported surface toward descriptive module-local names
- downstream code that referenced `rc.1`-era alias symbols or the short-lived numeric error names
  must update
- migration guide:
  - [../guides/migrating-from-0.1.0-rc.1.md](../guides/migrating-from-0.1.0-rc.1.md)

## Compatibility Notes

- Zig toolchain floor for this RC line remains `0.15.2`
- optional I6 exports remain build-flag gated
- wire formats, ownership posture, and the kernel-vs-SDK boundary are unchanged by this RC

## Docs And Examples

- updated the migration guide for downstream callers moving forward from `rc.1`
- updated the public style guide to prefer descriptive names inside module namespaces
- updated release-facing verification docs to include `zig build lint`

## Verification

- `zig build lint`
- `zig build test --summary all`
- `zig build`

## Upgrade Guidance

- if you depend on `noztr-core`, review [../guides/migrating-from-0.1.0-rc.1.md](../guides/migrating-from-0.1.0-rc.1.md)
- refresh any generated symbol indexes or local wrappers so they pick up the current descriptive
  error names
- use `rc.2` as the current pre-`1.0.0` baseline, not `rc.1`
